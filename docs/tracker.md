## Snapshot

**Phase 1 — Backend Foundation** (entered 2026-07-03, per root `CLAUDE.md`). **The backend build plan (`docs/architecture/backend-build-plan-v1.md`) is complete through Phase 6, and Phase 7 (Integration) is underway:** all five engines built and fixture-tested + the pipeline orchestrator + API v1, and now the **public Goal Simulator is live against the real engine** (`POST /simulations`, anonymous, no DB, no LLM key — structured intent skips Stage 1, narration is template-fallback). `pipeline/run.py` runs goal text → full Recommendation Package with **no DB and no LLM key required**; the determinism invariant is a green standing test. **Auth + persistence are LIVE end-to-end** (Supabase Auth + FastAPI JWT + live Supabase Postgres via the transaction pooler). A user logs in with Google → `api/auth.py` verifies the token (fails closed, `alg:none`/confusion blocked, `exp`/`aud`/`sub` required) → the session-aware nav shows them → the simulator's "Save this goal" button hits `POST /goals/recommendation/save` (`persist=True`) → the full lineage chain writes under their real `auth.users` id, and the endpoint reports `persisted` so the UI never claims a save that didn't land. Proven against the live DB with a real Google account. The signed-in surface is now a real dashboard: an app shell (sidebar/top bar/auth gate) whose home lists saved goals with a focus-goal hero, each opening into its full stored strategy (`GET /goals/{id}`, pure persisted-artifact reconstruction), with in-app goal creation at `/goals/new`. What's NOT yet built (follow-ups, not blockers): free-text goal entry in the UI; an automated real-DB round-trip test; score-breakdown/verdict persistence for saved-goal detail parity. Open risks: (a) ~60% of seed rows still carry `needs_verification: true`; (b) no LLM API key in `.env` yet (narration is template-fallback; D-5 streaming deferred until a live key is measured); (c) **Supabase DB password transited chat during debugging — rotate when convenient** (user has deferred); (d) snapshot loads once and caches — a transient cold-pooler refusal at first request serves 500s until restart (retry is a noted follow-up).
