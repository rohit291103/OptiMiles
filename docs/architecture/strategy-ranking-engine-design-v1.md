# 1. Purpose

## Objective

The Strategy Ranking Engine (SRE) evaluates every candidate strategy generated by the Strategy Generation Engine and determines which strategy delivers the highest overall value for a specific user.

Unlike the Strategy Generation Engine, which focuses on generating all feasible strategies, the Strategy Ranking Engine focuses on evaluating trade-offs and selecting the most appropriate strategy based on user context, business priorities and optimization objectives.

The engine produces one primary recommendation while preserving alternative strategies for comparison.

---

# 2. Scope

## In Scope

- Evaluate candidate strategies
- Compare strategy trade-offs
- Respect user constraints
- Apply user preferences
- Rank strategies
- Select primary recommendation
- Preserve ranked alternatives

## Out of Scope

- Generate strategies
- Calculate rewards
- Discover opportunities
- Simulate timelines
- Explain recommendations

---

# 3. Architecture Decisions

| ID | Decision |
|---|---|
| AD-01 | Ranking follows a multi-objective optimization philosophy. |
| AD-02 | Existing portfolio utilization is preferred whenever meaningful. |
| AD-03 | Hard constraints are never violated. |
| AD-04 | Preferences influence ranking but do not invalidate strategies. |
| AD-05 | One primary strategy is returned while preserving alternatives. |
| AD-06 | Every ranking decision must be explainable. |

---

# 4. Responsibilities

The Strategy Ranking Engine is responsible for:

- Evaluating candidate strategies
- Comparing strategy quality
- Resolving trade-offs
- Selecting the optimal strategy
- Producing a ranked strategy list

The engine is not responsible for creating or modifying strategies.

---

# 5. Ranking Philosophy

The engine does not optimize a single metric. Instead, it balances multiple objectives to maximize overall user value.

Primary objectives include: goal completion time, portfolio utilization, annual fee impact, strategy complexity, expected reward value, user preferences.

The selected strategy should represent the best overall balance rather than the highest score in any single dimension.

---

# 6. Ranking Dimensions

| Dimension | Description |
|---|---|
| Goal Completion Time | Time required to achieve the goal |
| Existing Portfolio Utilization | Preference for using current cards before recommending new ones |
| Reward Value | Overall reward potential |
| Cost | Annual fees and additional expenses |
| Strategy Complexity | Number of cards, transfers and execution steps |
| User Preferences | Desired trade-offs such as simplicity or speed |
| User Constraints | Mandatory conditions that cannot be violated |

*(Repo mapping: these correspond to the `score_breakdown` sub-scores in system-execution-flow-v1.md Stage 9 — goal_achievement, efficiency, cost, simplicity, portfolio_utilization, risk — with hard constraints filtering before scoring and preferences (DP-07) modulating weights, never overriding constraints.)*

---

# 7. Ranking Workflow

```
Candidate Strategy Set
        │
        ▼
Validate User Constraints
        │
        ▼
Evaluate Ranking Dimensions
        │
        ▼
Resolve Trade-offs
        │
        ▼
Rank Candidate Strategies
        │
        ▼
Select Primary Strategy
        │
        ▼
Publish Ranked Strategy List
```

# 8. Inputs

- Candidate Strategy Set
- User Profile
- Goal
- User Preferences
- User Constraints
- Business Rules

---

# 9. Outputs

## Primary Strategy

The highest-ranked strategy.

## Ranked Strategy List

All candidate strategies sorted by overall suitability.

## Alternative Strategies

Valid strategies that were not selected but may better suit different user preferences.

## Ranking Metadata

Supporting information explaining why a strategy ranked above another.

---

# 10. Business Rules

- **BR-01** Hard constraints must never be violated.
- **BR-02** Strategies using the existing portfolio should be preferred unless another strategy provides significant additional value.
- **BR-03** User preferences influence ranking but do not invalidate otherwise feasible strategies.
- **BR-04** Strategies with excessive complexity should only rank higher when they provide meaningful improvements.
- **BR-05** Every ranking decision must be explainable.
- **BR-06** The engine must always return one primary strategy.
- **BR-07** Alternative strategies must remain available for future re-ranking.
- **BR-08** Ranking must remain deterministic when provided with identical inputs.

---

# 11. Relationships

- **Strategy Generation Engine** — provides candidate strategies.
- **Reward Opportunity Engine** — provides opportunity metadata referenced by candidate strategies.
- **Simulation Engine** — provides projected timelines and outcome metrics used during ranking.
- **Recommendation Engine** — consumes the selected strategy and ranked alternatives to generate the final recommendation.

*(Repo mapping: "Strategy Generation Engine" is `optimization/strategies.py`; "Reward Opportunity Engine" is `valuation/opportunities.py`; both inside their respective five-CLAUDE.md engines, not standalone.)*

---

# 12. Failure Handling

| Scenario | Engine Behaviour |
|---|---|
| No candidate strategies | Return failure to Recommendation Workflow |
| Duplicate strategies | Merge equivalent strategies |
| Missing preference data | Apply default ranking behaviour |
| Conflicting constraints | Request clarification from upstream workflow |
| Invalid simulation data | Rank using available validated information only |

---

# 13. Assumptions

- Candidate strategies are valid.
- Simulation outputs are available when required.
- User constraints are already collected.
- Reward calculations are deterministic.

---

# 14. Constraints

- The engine cannot generate strategies.
- The engine cannot calculate rewards.
- The engine cannot personalize beyond provided preferences.
- The engine cannot modify strategy content.

---

# 15. Future Extensions

Adaptive ranking based on user behaviour, AI-assisted trade-off reasoning, multi-goal ranking, household-level optimization, explainable scoring visualization, learning-based ranking improvements.
