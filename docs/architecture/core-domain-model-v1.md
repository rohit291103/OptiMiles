# 1. Purpose

The Core Domain Model defines the fundamental business entities of OptiMiles and the relationships between them. It establishes a common business vocabulary used consistently across product documentation, engineering design, APIs, database schema, and user experience.

This document is implementation-independent and focuses solely on the business domain.

# 3. Business Entities

## 3.1 User
The person using OptiMiles. Owns: User Profile, Goals, Recommendations.

## 3.2 User Profile
All information required to personalize recommendations: existing credit cards, monthly spending, spending categories, reward balances, airline/hotel memberships, user preferences, constraints. Used by: Portfolio Analysis, Strategy Generation, Ranking.

## 3.3 Portfolio
Every reward asset currently owned by the user: credit cards, reward balances, loyalty programs.
Example: HDFC Infinia + HSBC TravelOne + 48,000 Reward Points + 12,000 KrisFlyer Miles.

## 3.4 Credit Card
An individual reward-earning instrument. Properties: issuer, network, annual fee, reward program, reward rules, transfer partners, milestone benefits, eligibility rules.

## 3.5 Spending Profile
How the user spends money every month: dining, grocery, fuel, travel, utilities, shopping, insurance, others. Used to estimate future reward accumulation.

## 3.6 Goal
The travel objective. Examples: Singapore Airlines Business Class, Marriott stay, lounge access. Properties: goal type, destination, origin, timeline, cabin class, hotel, passengers.

## 3.7 Reward Requirement
What is required to achieve the goal: target reward program, required reward currency, required miles/points, taxes/fees, redemption assumptions.

## 3.8 Reward Opportunity
An opportunity to maximize reward earning. Examples: SmartBuy voucher purchase, HSBC travel portal, dining multiplier, insurance milestone, festival bonus. Contains: eligible card, spend category, multiplier, conditions, expected reward.

> **MVP note:** portal/voucher/promotion-type opportunities require Merchant and Promotion knowledge (see [reward-knowledge-engine-spec-v1.md](reward-knowledge-engine-spec-v1.md) §6). The MVP schema models card × category × transfer-path opportunities first; portal/promotion opportunities are a post-MVP data extension.

## 3.9 Strategy
A complete execution plan enabling a user to achieve a travel goal through an optimized combination of reward earning, spend allocation, transfers, and redemption. Contains: card portfolio, spend allocation, reward opportunities, transfer plan, timeline, milestones, expected rewards. The primary output generated by the recommendation pipeline.

## 3.10 Recommendation
The final strategy presented to the user: strategy + summary + explanation + timeline + action items + alternative strategies.

# 4. Entity Relationships

- User owns User Profile; User owns Goals
- User Profile contains Portfolio; Portfolio contains Credit Cards and Reward Balances
- Spending Profile belongs to User Profile
- Goal produces Reward Requirement
- Reward Requirement identifies Reward Opportunities
- Reward Opportunities are combined into Strategies
- Strategy satisfies Goal
- Recommendation presents Strategy

---

# 5. Information Flow

```
User → Goal → Reward Requirement → User Profile → Portfolio
     → Reward Opportunities → Strategy → Recommendation
```

# 6. Repo Mapping (added in repo mirror)

How the domain vocabulary maps onto the execution pipeline objects ([system-execution-flow-v1.md](system-execution-flow-v1.md) §4) and the database schema ([db-schema-v1.md](db-schema-v1.md)):

| Domain entity | Pipeline object | DB home |
|---|---|---|
| Goal | `TravelGoal` (Stage 2) | `user_goals` |
| Reward Requirement | `RewardRequirement` (Stage 3) | denormalized to `user_goals.target_miles` |
| User Profile / Portfolio / Spending Profile | inside `PlanningContext` (Stage 4) | `users`, `user_cards`, `simulation_line_items` |
| Reward Opportunity | `RewardOpportunity` in `OpportunitySet` (Stage 5) | **derived, never stored** (RKE spec AD-05) |
| Strategy | `CandidateStrategy` (Stage 7) | winning + alternates in `simulation_results` JSONB |
| Recommendation | `FinalRecommendation` + narration (Stages 10–11) | `recommendation_outputs` |

Vocabulary discipline: when introducing a new domain term or noticing inconsistent usage, run the `domain-modeling` skill rather than coining ad-hoc names.
